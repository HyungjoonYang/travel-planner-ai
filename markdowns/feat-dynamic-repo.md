# Dynamic Repo — GitHub-Native Project Management

## Goal

evolve 파이프라인이 GitHub의 Issues, Discussions, Labels, Milestones를 활용하여
레포를 **살아있는 프로젝트**처럼 운영한다. `backlog.md`를 진실의 원천(source of truth)으로 유지하되,
GitHub 기능은 **가시성과 커뮤니케이션** 레이어로 활용한다.

---

## Architecture

```
backlog.md (source of truth)
    ↕ sync
GitHub Issues (visibility + communication)
    ↕ link
GitHub PRs (code delivery)
    ↕ track
GitHub Milestones (phase progress)

GitHub Discussions (decisions + retrospectives)
```

## 2. Milestones

- 각 Phase를 GitHub Milestone으로 생성
- Milestone description = Phase 설명
- Issue를 Milestone에 연결 → 진행률 자동 추적

## 3. Issues — 태스크 트래킹

### Architect Agent 변경
- 새 태스크를 `backlog.md`에 추가할 때 **동시에 GitHub Issue 생성**
- Issue body:
  ```markdown
  ## Task
  <태스크 설명>

  ## Spec Reference
  <markdowns 링크>

  ## Files
  <변경 대상 파일 목록>

  ## Done Criteria
  <완료 기준>

  ---
  🤖 Auto-generated by Architect Agent
  ```
- Label: `feature`/`bug`/`improvement` + `phase-N`
- Milestone: 현재 Phase

### Coordinator Agent 변경
- 태스크 선택 시 해당 Issue에 **코멘트** 추가:
  ```
  🧠 Coordinator: 이 태스크를 Run #<N>에 배정했습니다.
  ```
- Issue를 `In Progress` 라벨로 변경 (선택사항)

### Reporter Agent 변경
- QA pass → **Issue close** + PR에 `Closes #<issue>` 포함
- QA fail → Issue에 **실패 코멘트** 추가:
  ```
  🧪 QA Failed (Run #<N>): <실패 내역>
  다음 실행에서 재시도합니다.
  ```

## 4. Discussions — 의사결정 + 회고

### Category 구조
| Category | 용도 |
|----------|------|
| `Tech Decisions` | 기술 결정 논의 (기존 tech-decisions.md 보완) |
| `Retrospectives` | 장애 postmortem, 주간 회고 |
| `Ideas` | 새 기능 아이디어, 개선 제안 |

### Architect Agent
- 중요한 설계 결정 시 Discussion 생성
  - 제목: `[Decision] <결정 사항>`
  - Category: Tech Decisions
  - Body: 결정 내용, 대안, 선택 이유

### Reporter Agent
- 일일 요약을 Discussion으로도 게시 (주 1회 또는 Phase 전환 시)
  - 제목: `[Weekly] Phase N 진행 현황`
  - 태스크 완료 수, 테스트 수, 주요 변경사항

## 5. PR Enhancement

- `Closes #<issue>` 자동 포함 → 머지 시 Issue 자동 close
- Milestone 연결
- 관련 Discussion 링크

## 6. Error / Incident

- 테스트 실패 3회 연속 → **Bug Issue 자동 생성**
  - Label: `bug`, `blocked`
  - Body에 실패 내역, 관련 코드 경로 포함
